## 💡 BFF가 궁금하다

BFF를 위해 GraphQL를 적용했다는 대기업 기술 블로그 글을 읽었다. 아마 GraphQL이 REST API가 가지는 오버패칭과 언더패칭이라는 단점을 보완해주기 때문에 "프론트엔드의 요구에 맞춘 백엔드를 따로 둔다"는 BFF의 요구사항에 맞아떨어지기 때문에 채택된 것일듯 싶다.

그렇다면 궁금하다. 뭐든 방법론적인거, 또는 공식 문서에서 친절하게 알려주는 "된다", "됩니다"가 정확히 "어떻게 되는가"로 전환되기 위한 호기심을 해소하고 싶다. 다만, 실무가 아닌 개인 프로젝트에서 적용해보는 것이 "제가 이런걸 해봤습니다" 라는 말을 대변해줄지가 의심스럽지만, 조만간 BFF를 담당해줄 서버를 새로 만들어서 프론트엔드와 붙여보고 싶다. 인터페이스를 위한 레이어를 하나 만드는 느낌이려나.. 해보자 !

## 💡 Lighthouse 자동화

Url을 던져주면 Lighthouse 기반으로 분석 자동화해주는 애플리케이션이 있는지 궁금하다. 없으면 디자인, UX도 편리하게 해서 만들어서 무료로 배포하면 재밌겠다.

시중에는 `Calibre`, `Treo`가 있는데 전자는 유료, 후자는 안정성 문제가 있다고 한다.

어떻게 만들 수 있는가.

Google에서 제공해주는 Lighthouse 기반 분석 API인 PageSpeed Insights API라는게 있다. 페이지별 Performance, SEO, Accessibility 등 점수와 상세 피드백을 포함한 리포트를 보여주면 좋겠다는 생각.

게다가 해당 API는 일일 25,000 요청, 초당 60요청의 무료 API

```js
[ 사용자가 URL 입력 ] → [ 서버가 Headless Chrome에서 Lighthouse 실행 ] → [ 결과 분석 및 시각화 ]

//
```

위와 같은 프로세스가 될테고, 리포트 시각화는 `Recharts`나 `Chart.js`로 하면 되겠다.

## 💡 구조조정된 이들을 위한 서비스

미국에서 엄청난 수의 공무원들이 일자리를 잃었는데, 그들을 활용(?)해 .. 아니 그들의 니즈를 파악해서 특정 서비스를 만들면 즐거운 개발 경험이 될지도 모른다.

물론 이건 제대로 하려면, 전에 무슨 책에서 읽은 것처럼 수 차례의 인터뷰와 설문조사를 통해 범위를 좁혀가면서 아이템 방향성이 나오겠지만,, 그냥 번뜩 생각이 나서 까먹기 전에 메모.

연령층이 다양하기에 높은 연령층을 고려해 생각보다 복잡하지 않고 단순한 애플리케이션이면 좋을 것
